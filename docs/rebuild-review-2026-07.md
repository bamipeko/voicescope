## 1. 総評 — 結論: 「部分作り直し」

**継ぎ足し継続 / 部分作り直し / 全面再構成 のうち、「部分作り直し」を推奨する。**

具体的には:

1. **PC版（Electron + Express + React + sql.js）は作り直さない。凍結保守にする**
   - v0.18.0まで作り込まれた動くプロダクト。捨てる理由がない
   - ただし新機能追加も止める。重大バグ修正のみ
2. **スマホ版は、既存の Capacitor 路線（mobile/ scaffold）を破棄し、Expo (React Native) で新規構築する**
   - 理由は後述。「client/ を90%再利用できる」という Capacitor 採用時の前提が実態と合っていない
3. **Cloudflare Worker（おまかせモードの中核）はコード完成済み。そのまま採用し、最初の1週間でデプロイする**
4. **ローカルAI処理は「エンジンを1本化」して再設計する**（faster-whisper の Python 経路は廃止）

### なぜ「継ぎ足し継続」ではないのか

現行アーキテクチャの本質的な問題は、**アプリの頭脳がすべて Express サーバ（server/ 約8,000行）に入っている**こと。client/ は `lib/api.js` 経由で HTTP を叩くだけの薄い画面層で、DB・ファイル・パイプライン・AIプロバイダの全ロジックがサーバ側にある。

スマホには Node.js ランタイムがない。つまり Capacitor で client/ を包んでも、**アプリの本体である server/ は1行も持っていけず、全部クライアント側に書き直しになる**。2026-05 のロードマップの TODO 4・5（「DB移行: server のロジックを capacitor-sqlite に置換」「Filesystem 抽象化」）が2ヶ月止まっているのは偶然ではなく、これが実質「作り直し」だから。継ぎ足しでスマホ対応に到達するルートは存在しない。

### なぜ「全面再構成」でもないのか

- PC版は現に動いていて、Bami自身の日常利用データ（録音38件・文字起こし50件）を支えている。壊す理由がない
- worker/（おまかせモードのAPIプロキシ）は設計・実装とも健全で、そのまま使える
- プロダクト設計（プラン構成、テンプレート資産、パイプライン設計、要約プロンプト、UXの知見）は全部生きている
- 捨てるべきは「実装の置き場所」であって、プロダクトそのものではない

### 耳の痛い指摘（先に言う）

**「スマホでローカル保存できれば他には負けない」は、そのままでは楽観的すぎる。**

- Pixel 標準のレコーダーアプリは端末内文字起こしを標準搭載しており、iPhone もボイスメモ/メモの文字起こしを OS 機能として搭載してきている（日本語対応も拡大中）
- PLAUD / Notta / Otter などクラウド系も強い。「ローカル保存」単体は差別化の柱として細い

勝ち筋はもっと複合的に定義すべき（§7 事業・差別化で詳述）:
「**データを人質に取らない**（全データがSQLite+音声ファイルでエクスポート自由）」×「**要約テンプレート資産**（タスク抽出・感情分析など自分のプロンプト）」×「**PC/スマホ横断のローカルアーカイブ**」×「**非エンジニアの身内にそのまま渡せるおまかせモード**」。この組み合わせなら個人開発でも守れるニッチになる。

## 2. 推奨アーキテクチャ

### 全体像

```
┌─────────────────────────────────────────────────────┐
│ packages/core  (TypeScript・プラットフォーム非依存)     │
│  ・データモデル + SQLiteスキーマ + マイグレーション      │
│  ・パイプライン状態機械（録音→文字起こし→要約→タグ）     │
│  ・AIプロバイダクライアント                             │
│    (Deepgram/OpenAI/Gemini/Claude/Grok/Ollama/Worker)  │
│  ・プロンプト/テンプレート定義                          │
│  ・VoiceScopeアーカイブ形式 (エクスポート/インポート)     │
│  ※ DBアクセスとファイルI/Oはアダプタinterface経由        │
└───────────────┬─────────────────────┬───────────────┘
                │                     │
   ┌────────────▼───────────┐   ┌─────▼──────────────────┐
   │ apps/mobile (新規)      │   │ 既存PC版 (凍結保守)      │
   │ Expo / React Native    │   │ Electron+Express+sql.js│
   │ ・ネイティブ録音         │   │ ・現状のまま動かし続ける  │
   │   (Foreground Service) │   │ ・重大バグ修正のみ        │
   │ ・expo-sqlite          │   │ ・将来 core 採用へ置換も  │
   │ ・オンデバイスSTT        │   │   可（急がない）         │
   │   (whisper.rn 等)      │   └─────┬──────────────────┘
   └────────────┬───────────┘         │
                │                     │
        ┌───────▼─────────────────────▼────────┐
        │ 処理バックエンド（ユーザーが選ぶ3系統）    │
        │ ① 端末内: オンデバイスSTT / (将来)端末内LLM│
        │ ② 自宅: PC/NAS の Ollama・whisper (HTTP) │
        │ ③ おまかせ: Cloudflare Worker (managed)  │
        │    └ own-key モードは各社API直叩き        │
        └──────────────────────────────────────┘

  データの置き場所（常にローカルが正）:
  ・スマホ: アプリ領域の SQLite + 音声ファイル
  ・PC:     %APPDATA%\VoiceScope\ (現状のまま)
  ・交換:   VoiceScopeアーカイブ(zip) で バックアップ/移行/PC連携
```

### 設計原則

1. **「ローカル保存」は無条件、「ローカル処理」は段階式**
   - 録音・文字起こし・要約データは常に端末内 SQLite + ファイルに保存（これは絶対）
   - 処理（STT/LLM）は 端末内 → 自宅PC/NAS → おまかせクラウド の3系統から選択・フォールバック
2. **コアロジックは一度だけ書く**: packages/core にまとめ、スマホが第一の利用者。PC版の置き換えは急がない
3. **同期エンジンは作らない（最初は）**: 端末間連携は「アーカイブのエクスポート/インポート」で始める。双方向同期は需要が確認できてから
4. **モバイルUIは新規設計**: 現行 client/ はサイドバー型のデスクトップUI（RecordingDetail.jsx 1,962行）。スマホに移植するものではない

## 3. 技術選定比較（スマホアプリ化）

### 比較表

| 観点 | Capacitor（現行案） | **React Native / Expo（推奨）** | Flutter | PWA | ネイティブ (Kotlin/Swift) |
|---|---|---|---|---|---|
| 既存資産の再利用 | UI再利用は理論上可だが、デスクトップUIはスマホに不適。**サーバロジックはどのみち全書き直し** | UIは書き直しだが React の知見・TSコア共有が効く | ほぼゼロ（Dart）。TSコアも共有不可 | client/ ほぼそのまま | ゼロ |
| 長時間・バックグラウンド録音 | WebView の MediaRecorder は長時間録音に構造的リスク。結局ネイティブプラグイン依存（Cap-go audio-recorder 等はあるが選択肢が薄い） | ネイティブ録音 + Foreground Service。実績ライブラリ複数（expo-audio、@siteed/expo-audio-studio 等） | record + foreground service 系で可 | **不可**（画面オフで停止。致命的） | 最も確実 |
| 録音ファイル形式 | WebView録音は webm → ffmpeg変換問題を引きずる | **ネイティブ録音で最初から m4a/wav。現行の webm 変換エラー問題が構造的に消える** | 同左 | webm 問題継続 | 同左 |
| オンデバイスSTT | 成熟した whisper 系プラグインなし。自作ネイティブプラグインが必要 | **whisper.rn（whisper.cpp ベース、実績あり）**。sherpa-onnx 系も選択肢 | sherpa-onnx の Flutter 対応あり | WASM whisper はスマホでは非実用 | whisper.cpp / sherpa-onnx 直組み込み |
| ローカルDB | @capacitor-community/sqlite（コミュニティ維持） | **expo-sqlite（Expo公式維持）** / op-sqlite | sqflite / drift | OPFS/IndexedDB は**OSに消される可能性**（永続保証なし） | Room / GRDB |
| iOS 展開 | 可 | **可（同一コード。EASクラウドビルドで Mac 不要に近い）** | 可 | 可(制約大) | 別コードベース必要 |
| AI駆動開発（Codex）との相性 | 中 | **高（エコシステム最大級、情報量多い）** | 高 | 高 | 中 |
| 開発速度（今回の前提で） | 「速い」の前提が崩壊済み | 中〜速 | 中 | 最速だが要件を満たせない | 遅い |

### 判定: **Expo (React Native) を採用**

決め手は3つ:

1. **Capacitor の優位性だった「90%再利用」が実態と合わない。** サーバ側8,000行は再利用不可、デスクトップUIはスマホUXに不適。残る再利用対象（プロバイダロジック・スキーマ・プロンプト）は TS コアに移す作業であり、それは RN でも同じだけ効く
2. **このアプリの生命線は「録音の信頼性」と「オンデバイスSTT」で、どちらもネイティブ領域。** RN には whisper.rn・expo-audio・expo-sqlite という実績部品が揃っており、WebView 越しに戦う Capacitor より土台が厚い
3. **ネイティブ録音により、現行ローカル処理エラーの主犯の一つ（webm形式→ffmpeg変換）が構造的に消える**

Flutter は次点（Sumika で環境共有できる利点はある）が、JS/TS 資産とコア共有できない点で不利。PWA はバックグラウンド録音不可・ストレージ永続保証なしの2点で主戦場には不適格（既存 Docker/ブラウザ版は閲覧用として残せば十分）。

### リスクと打ち消し

- **RN は UI をゼロから書く** → モバイルMVPは画面3〜4枚に絞る（§6）。デスクトップの1,900行詳細画面を再現しない
- **whisper.rn の Android 性能は端末依存**（iOS比で遅い報告あり）→ 検証週で実機計測。遅ければ「オンデバイスは短い録音のみ、長尺はクラウド/自宅PC」の自動振り分けで対応
- **Expo/ネイティブモジュールのビルド詰まり** → EAS Build（クラウドビルド）を最初から使い、ローカル Android Studio 依存を減らす

## 4. データ保存・移行設計

### 方針: 「既存データは動かさない」が最も安全

- **PC の `voicescope.db` + 音声はそのまま PC に残す。** PC版を凍結保守で動かし続けるので、強制移行が発生しない = データ喪失リスクが構造的にゼロ
- スマホ版は新規データから開始。PCデータの取り込みは「アーカイブインポート」機能として後続フェーズで提供
- なお 2026-06-24 に全文字起こしの txt 一括エクスポート済み（`exports/transcripts_20260624094647`）。着工前に DB + audio フォルダの手動バックアップも1回取る（コピーするだけ）

### スマホ側の保存構成

| データ | 置き場所 | 理由 |
|---|---|---|
| メタデータ・文字起こし・要約・テンプレ・タグ | **expo-sqlite（端末内SQLiteファイル）** | PC版と同じ論理スキーマを踏襲。信頼性・検索・移行性 |
| 音声ファイル | アプリのドキュメント領域（m4a推奨） | OSに消されない領域。DB にはパスのみ |
| APIキー・設定 | expo-secure-store（OSのキーチェーン） | 平文保存禁止ルール準拠 |

- **論理スキーマは PC 版と揃える**（recordings / transcriptions / summaries / templates / tags）。現行 schema.sql の設計は健全で、そのまま core のスキーマ定義の出発点にできる
- sql.js 特有の地雷（`db.export()` で rowid リセット等）は expo-sqlite では発生しない。ネイティブSQLiteなので毎回全DB書き出しも不要になる

### VoiceScope アーカイブ形式（v1で定義する・これが背骨）

```
voicescope-archive-YYYYMMDD.zip
├─ manifest.json      (形式バージョン、端末、件数)
├─ records.jsonl      (recordings+transcriptions+summaries+tags を1行1録音で)
└─ audio/             (音声ファイル実体)
```

この1つの形式で「バックアップ」「端末移行」「スマホ⇄PC連携」「復元」を全部まかなう。共有シート経由で Drive/NAS/PC どこにでも出せる。PC版に「アーカイブ書き出し/読み込み」を1機能だけ後付けすれば、既存 `voicescope.db` の中身もスマホに持ち込める（これが唯一PC側に加える変更で、優先度は低くてよい）

### バックアップ・復元

- Phase R1: 手動エクスポート（共有シート → Drive / NAS / PC）
- Phase R2: 「Wi-Fi 接続時に NAS/自宅PCへ自動バックアップ」（単方向コピー。同期ではない）
- 復元 = 新端末でアーカイブをインポート。以上

## 5. ローカルAI設計 — 「エラーが起きやすい」の根治

### 現状の問題の構造

ローカル処理が壊れやすいのはバグの集積ではなく設計の問題。原因は:

1. **STTエンジンが3系統並存**（whisper.cpp バイナリDL / faster-whisper Python サブプロセス / クラウド）で、それぞれ別のセットアップ・失敗モードを持つ
2. **Python 依存**（faster-whisper）は非エンジニアには最悪の依存（環境構築・パス・バージョン地獄）
3. **webm → wav 変換**（ブラウザ録音の副産物）が変換エラーの温床
4. 失敗時に「何が悪いか」がユーザーに見えず、回復導線がない

### 再設計: 役割分担マトリクス

| 処理 | スマホ | PC | 自宅サーバ(PC/NAS) | クラウド |
|---|---|---|---|---|
| 録音・保存 | ◎ 常に端末内 | ◎ 常に端末内 | − | ✕（保存はローカルが正） |
| 文字起こし | ○ whisper.rn（短〜中尺） | ○ whisper.cpp 1本化 | ○ 長尺の逃がし先（後続） | ◎ Deepgram（既定・高品質） |
| 要約 | △ 端末内LLMは当面見送り※ | ○ Ollama エンドポイント | ◎ RTX 5070 Ti の Ollama | ◎ Worker おまかせ / own-key |
| 画像生成 | ✕ | ✕ | ✕ | ◎ Worker 経由 gpt-image-2 のみ |

※端末内LLM要約: Android は ML Kit GenAI（Gemini Nano、要約APIは日本語対応）、iOS 26 は Foundation Models が現実化しているが、**対応端末が限られる**。「対応端末なら追加ボタンが出る」opt-in 機能として後続フェーズで載せる。コア導線には置かない。

### 具体的な再設計ポイント

1. **faster-whisper（Python経路）は廃止。** PCのローカルSTTは whisper.cpp 1本
2. **スマホのローカルSTTは whisper.rn 1本**（検証週で sherpa-onnx + 日本語特化モデル ReazonSpeech 系、kotoba-whisper の ggml 版も比較候補に入れる）
3. **モデル管理を「状態が見える」UIに**: モデルDL済み/未DL/破損 をアプリが自己診断し、ワンタップ再取得。「silent fail」を仕様として禁止
4. **フォールバックを仕様に組み込む**: ローカルSTT失敗 → その場で「クラウドで再実行しますか？」を提示（自動ではなく提案。通信させたくないユーザーの意思を尊重）
5. **ネイティブ録音で m4a 直保存** → 変換工程そのものを消す
6. **要約のローカル経路は「ローカルエンドポイント（Ollama互換URL）」方式を継続**。既存 `localEndpoint.js` の5プリセット設計は良い設計なので core に引き継ぐ。非エンジニアには「おまかせプラン」が答えであり、Ollama設定は上級者向けと割り切って導線を分ける

## 6. 実行ロードマップ

### Phase R0: 検証週（1週間）— ゲート付き

| # | 検証項目 | 合格基準 |
|---|---|---|
| 1 | **Worker 本番デプロイ**（wrangler deploy + secrets + code seed） | /verify → JWT → chat/completions が通る。既存Electron版のおまかせモードが動く |
| 2 | **Expo 録音 PoC**（実機 Android） | 画面オフ・アプリ裏で60分録音が完走、m4a が保存される。着信で壊れない |
| 3 | **オンデバイスSTT PoC** | whisper.rn で日本語10分音声を実用時間で文字起こし（目安: 実時間の2倍以内）。精度を Deepgram と並べて目視比較 |
| 4 | expo-sqlite + ファイル保存の基本動作 | スキーマ作成・CRUD・再起動後の永続確認 |

**ゲート判定:** 2 が不合格なら @siteed/expo-audio-studio → 自作ネイティブモジュールの順で代替。3 が不合格でも中止ではなく「ローカルSTTは短尺のみ、長尺はクラウド/自宅PC」に降格してMVP続行。

### Phase R1: モバイルMVP（2〜4週間）— 自分用

画面は4枚だけ:
1. **録音**（1タップ開始・停止、録音中ハイライトボタン、バックグラウンド継続）
2. **一覧**（検索、状態バッジ）
3. **詳細**（再生+文字起こし表示、要約実行・表示、テンプレ選択）
4. **設定**（処理モード: おまかせ/own-key/ローカル、モデル管理、エクスポート）

機能パイプライン: 録音 → 端末内保存 → 文字起こし（選択式）→ 要約（テンプレ）→ アーカイブエクスポート。
テンプレートは PC 版の資産（タスク抽出・感情分析等のプロンプト）を初期データとして同梱。

**ゲート:** Bami 自身が2週間、日常の録音をスマホ版だけで回せる。

### Phase R2: 友人ベータ（+2〜4週間）

- おまかせコード発行 → 初回起動ウィザード（コード入れるだけ）
- 自動バックアップ（Wi-Fi時にNAS/PCへ）
- 導入ガイド（インストール→録音まで、スクショ付き1枚）
- **ゲート:** 非エンジニア2〜3人が、セットアップ質問ゼロで1週間使える

### Phase R3: 拡張（需要を見て）

- PC版に「アーカイブ書き出し/読み込み」追加（既存データのスマホ取り込み）
- Play ストア クローズドテスト配布（サイドロード規制が段階的に強まる流れへの備え）
- iOS ビルド（EAS。Expoなら同一コード）
- 対応端末向け端末内LLM要約（ML Kit GenAI / Apple Foundation Models）
- 課金（Stripe）は**実ユーザーの支払い意思が確認できてから**。それまではトライアルコード運用

### 最初に捨てる/凍結するもの

- `mobile/`（Capacitor scaffold）→ 削除（git履歴に残る）
- faster-whisper Python 経路 → 廃止
- モバイルMVPから除外: インフォグラフィック生成 / CrossAsk / フォルダ / Archive・Trash 3層 / 会議アプリ検知 / Discord RPC / 話者ラベル編集（Phase R3以降で個別判断）
- PC版の新機能開発 → 全面凍結（重大バグのみ）
- Docker/ブラウザ版 → レガシー扱いで放置（消さない）

## 7. 事業・差別化の評価

### 率直な市場認識

- 「録音→文字起こし→要約」は激戦区。クラウド系（Notta/Otter/PLAUD）はもちろん、**OS標準機能（Pixelレコーダー、iPhoneボイスメモ）が端末内文字起こしを無料で配りつつある**のが最大の脅威
- したがって「スマホ×ローカル」を唯一の柱にしない。柱は4本束ねる:

| 差別化軸 | 対クラウド競合 | 対OS標準機能 |
|---|---|---|
| データ完全所有（SQLite+音声+アーカイブでいつでも全持ち出し） | ◎ 彼らはクラウド囲い込み+サブスク | ○ OS標準は書き出しが貧弱 |
| 要約テンプレート資産（タスク抽出・感情分析・自分のプロンプト） | ○ | ◎ OS標準にはない |
| PC/スマホ横断のローカルアーカイブ | ◎ | ◎ |
| 身内・コミュニティ配布（おまかせコード、設定ゼロ導入） | ◎ 個人が身内に配れる価格・形態ではない | − |

### 現実的な位置づけ

これは「Otter に正面から挑む事業」ではなく、**「Bami自身とコミュニティの録音資産を完全所有するためのプロダクトで、身内課金（Pro ¥980/Heavy ¥2,480）が回れば上出来」**という規模感で運営するのが正しい。その前提なら:

- 固定費はほぼ Worker の API 従量のみ（コード発行数で統制可能）
- 撤退条件も軽い（最悪、自分用ツールとして残るだけで損失は開発時間のみ）
- 逆に言うと、**マーケティングや汎用SaaS化への投資は現時点では不当**。R2ゲート（非エンジニアが質問ゼロで使える）を通ってから考える

## 8. 既存コードの残す/捨てる（大きな判断のみ)

| 資産 | 判断 | 備考 |
|---|---|---|
| PC版一式（client/server/electron） | **残す（凍結保守）** | 動くプロダクト。作り直し対象ではなく「維持する旧本線」 |
| worker/ 一式 | **残す・即デプロイ** | 新アーキテクチャでもそのまま managed バックエンド |
| server/services/summary/*・transcription/*（プロバイダロジック） | **ロジックを core へ移植** | コピペではなく TS で書き直すが、仕様の教科書として価値大 |
| server/db/schema.sql | **残す（coreスキーマの原型）** | 設計は健全 |
| テンプレート/プロンプト資産（DB内） | **残す・モバイルに同梱** | 最重要の差別化資産 |
| client/src/lib/{platform,storage,localEndpoint}.js | 設計思想を core に引き継ぐ | localEndpoint の5プリセット設計は特に良い |
| mobile/（Capacitor scaffold） | **捨てる** | cap init 前で実装ゼロ。損失なし |
| faster-whisper Python 経路 | **捨てる** | 非エンジニア配布と両立しない |
| sql.js | 新規コードでは使わない | PC版は凍結中そのまま。次にPC版を本格改修する時 node:sqlite 等へ |
| Docker/ブラウザ版 | 凍結（消さない) | NAS閲覧用として一応残る |
| state/ ログ運用・地雷マップ | **残す** | 引き継ぎ資産として優秀 |

## 9. 確認事項への回答（2026-07-03 Bami回答・確定）

1. **Android 実機 = OPPO Pad Air (OPD2102A)**: タブレット（Snapdragon 680 / 4GB RAM / 最大 Android 13 / Wi-Fiのみ）
   - 録音メカニクス（Foreground Service・保存・電池）の検証には使えるが、低スペックなのでオンデバイスSTTの性能判定はこの端末を基準にしない
   - 「ポケットの中のスマホで録る」検証にはならない → 日常持ち歩くスマホの機種は引き続き確認（TBD）
2. **iOS もやる（iPhone 11 実機あり）**: Expo 採用の判断をさらに補強。ただし実機配布には Apple Developer Program（年額約1.5万円）がほぼ必須 → R0 で段取り調査、加入判断は R1 前
3. **録音尺は3時間必要**: オンデバイスSTTは「短いメモ用」に降格が濃厚。長尺ローカル処理の本命は**自宅PC(RTX 5070 Ti)オフロード**とし、優先度を R3 → R2 に引き上げ。クラウド（Deepgram、3時間≈$0.8）が当面の既定
4. **PC版ローカル処理のエラーは原因特定済み**: whisper.cpp が large-v3（3.1GB）を CPU のみ・beam 5 で実行し、66分録音の途中で code null（メモリ枯渇濃厚）× 6回（`server-2026-06-24.log`）。GPU 未使用。Ollama 側のエラーは無し
   - 「凍結保守の重大バグ」として応急修理を R0 に組み込み（長尺チャンク分割ほか）。詳細は `docs/briefs/phase-R0-brief.md` Scope B
